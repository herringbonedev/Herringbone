# Herringbone Auth API

The Herringbone Auth API provides authentication, user management, service identity, and platform authorization for the Herringbone platform.

The service issues JWT tokens for users and services and enforces scope based permissions.

Base route:

/herringbone/auth


# Authentication

Users authenticate with email and password.

Successful authentication returns a JWT access token.

Services authenticate using service tokens generated by a platform administrator.

All tokens contain scopes which determine what the caller is allowed to access.


# Scopes

Scopes represent platform level permissions.

Example scopes

| Scope           | Meaning                |
|-----------------|------------------------|
| *               | platform owner         |
| platform:admin  | platform administrator |
| logs:read       | read logs              |
| search:query    | execute searches       |
| incidents:read  | read incidents         |


# Bootstrap

The first user must be created using a bootstrap token.

Provide the token in the request header.

Header

x-bootstrap-token

Example

curl -X POST http://localhost:8000/herringbone/auth/register \
-H "x-bootstrap-token: YOUR_BOOTSTRAP_TOKEN" \
-H "Content-Type: application/json" \
-d '{
  "email": "admin@example.com",
  "password": "StrongPassword123"
}'

The first user automatically receives the * scope.


# User API


Create user

POST /herringbone/auth/register

Request

{
  "email": "user@example.com",
  "password": "StrongPassword123",
  "scopes": ["logs:read"]
}

Response

{
  "ok": true,
  "user_id": "mongodb_id",
  "scopes": ["logs:read"]
}


Login

POST /herringbone/auth/login

Request

{
  "email": "user@example.com",
  "password": "StrongPassword123"
}

Response

{
  "access_token": "jwt_token",
  "token_type": "bearer"
}


List users

GET /herringbone/auth/users


Update user scopes

POST /herringbone/auth/users/scopes

Request

{
  "email": "user@example.com",
  "scopes": ["logs:read", "search:query"]
}


Delete user

DELETE /herringbone/auth/users

Request

{
  "email": "user@example.com"
}


# Service API

Service accounts allow internal Herringbone services to authenticate.


Register service

POST /herringbone/auth/services/register

Request

{
  "service_name": "parser",
  "scopes": ["logs:read"]
}

Response

{
  "ok": true,
  "service_id": "mongodb_id",
  "service_name": "parser"
}


List services

GET /herringbone/auth/services


Create service token

POST /herringbone/auth/service-token

Request

{
  "service": "parser",
  "scopes": ["logs:read"]
}

Response

{
  "access_token": "jwt_token",
  "token_type": "bearer"
}


Delete service

DELETE /herringbone/auth/services/{service_name}


Remove service scopes

POST /herringbone/auth/services/scopes/remove

Request

{
  "service_name": "parser",
  "scopes": ["logs:read"]
}


# Scope Catalog

List available scopes

Health check

GET /herringbone/auth/readyz

# Security Implementation


Password hashing

User passwords are stored using PBKDF2 HMAC SHA256.

Configuration

PBKDF2 iterations

310000

Salt size

16 bytes

Password hashes are stored as

base64(salt + pbkdf2_hash)

Password verification recomputes the PBKDF2 hash and compares it using constant time comparison.


Secret storage

Secrets are loaded from container secret files.

User JWT secret

/run/secrets/jwt_secret

Service signing key

/run/secrets/service_jwt_private_key


User tokens

User tokens are signed using HS256.

Expiration

24 hours

Token payload

sub      user id
email    user email
scope    list of scopes
typ      user
iat      issued timestamp
exp      expiration timestamp

Example

{
  "sub": "user_id",
  "email": "user@example.com",
  "scope": ["logs:read"],
  "typ": "user",
  "iat": 1700000000,
  "exp": 1700086400
}


Service tokens

Service tokens are signed using RS256.

Expiration

365 days

Token payload

iss      herringbone-auth
sub      service id
service  service name
scope    scopes
typ      service
aud      herringbone-services
iat      issued timestamp
exp      expiration timestamp

Example

{
  "iss": "herringbone-auth",
  "sub": "service_id",
  "service": "parser",
  "scope": ["logs:read"],
  "typ": "service",
  "aud": "herringbone-services",
  "iat": 1700000000,
  "exp": 1731536000
}

Service tokens are signed using the private key located at

/run/secrets/service_jwt_private_key

Other services validate the token using the public key.


Token types

The typ field identifies the token type.

user
service
